# 📝 TIL (Today I Learned) - 알고리즘 문제 풀이 복습
## 1. 문제 개요

* 핵심: 연결된 컴포넌트(Network)의 개수를 구하는 그래프 탐색 문제.
* 알고리즘: BFS(Breadth-First Search) 활용.
* 복잡도:
* 시간: $O(N^2)$ (모든 노드를 방문하며 각 노드마다 인접 행렬의 $N$개 요소를 확인)
    * 공간: $O(N)$ (방문 체크 배열 및 큐 관리)

## 2. 언어별 구현 포인트 및 비교

| 언어 | 큐(Queue) 구현체 | 특징 및 피드백 |
|---|---|---|
| Java | ArrayDeque | LinkedList보다 캐시 지역성이 좋고 메모리 오버헤드가 적어 권장됨. |
| Python | collections.deque | popleft()를 통한 $O(1)$ 연산 지원. enumerate를 활용한 깔끔한 인덱스 추출. |
| Go | Slice 기반 커스텀 | 표준 큐 라이브러리가 없어 슬라이스 연산(queue[1:])이나 인덱스 포인터로 구현. |

## 3. 언어별 핵심 테크닉

* Java: 벤치마크(JMH)와 JUnit5를 활용한 검증 환경 구축. 인접 행렬을 직접 참조하여 불필요한 객체 생성을 최소화함.
* Python: while q:와 if not visited[i]: 조합으로 직관적인 탐색 코드 작성. $O(N^3)$처럼 보일 수 있으나 실제로는 모든 간선을 한 번씩만 확인하는 $O(N^2)$임을 확인.
* Go:
* make([]int, 0, n)으로 초기 용량을 설정하여 슬라이스 재할당 방지.
    * 메모리 관리: queue[1:] 방식의 메모리 누수 가능성을 인지하고, 성능 최적화를 위해 인덱스 포인터(head) 방식이 유리함을 배움.

## 4. 추가 인사이트

* 인접 행렬 vs 인접 리스트: 노드 수가 적고 간선이 많은(밀집 그래프) 문제 특성상, 별도의 리스트 변환 없이 입력받은 computers 행렬을 직접 사용하는 것이 공간/시간 면에서 효율적임.
* 메모리 효율: Go와 Java 모두 재사용 가능한 구조(배열, 슬라이스)를 루프 밖에서 선언하여 GC(Garbage Collection) 부담을 줄이는 것이 성능 최적화의 핵심.
