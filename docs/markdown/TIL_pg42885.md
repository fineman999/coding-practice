# 📝 TIL (Today I Learned) - 알고리즘 문제 풀이 복습

* 목표: 최소한의 구명보트를 사용하여 모든 사람 구출.
* 제약: 한 보트에 최대 2명, 무게 제한 limit 존재.
* 핵심 아이디어: 가장 무거운 사람을 보낼 때, 남은 사람 중 가장 가벼운 사람을 끼워 보낼 수 있는지 확인한다. (Greedy)

## 2. 언어별 구현 포인트 및 복습## 🐍 Python (사고의 흐름)

* 초기 시도: deque를 활용해 가벼운 순서대로 태움 → 실패 (2명 제한 및 최적 조합 미고려).
* 최종 해결: sort 후 popleft()와 pop()을 이용한 양방향 접근.
* 복잡도: $O(N \log N)$ 시간 / $O(N)$ 공간 (deque 생성 비용).

## 🐹 Go (효율성 극대화)

* 포인트: deque 대신 투 포인터(Left, Right)를 사용하여 추가 메모리 할당을 0으로 만듦.
* sort.Ints() 또는 slices.Sort()를 사용하여 원본 슬라이스 내에서 정렬.
* 복잡도: $O(N \log N)$ 시간 / $O(1)$ 공간.

## ☕ Java (성능과 구조)

* 포인트: Primitive Type인 int[]를 직접 다루어 속도 최적화.
* Arrays.sort()와 while (left <= right) 루프를 이용한 정석적인 투 포인터 구현.
* 테스트: JMH를 통한 마이크로 벤치마크와 JUnit5를 이용한 엣지 케이스 검증.
* 복잡도: $O(N \log N)$ 시간 / $O(1)$ 공간.

## 3. 공통 핵심 로직 (Pseudo Code)

1. 사람들의 몸무게를 오름차순으로 정렬한다.
2. 가장 가벼운 사람(left)과 가장 무거운 사람(right)을 가리키는 포인터를 설정한다.
3. left <= right인 동안 반복:
    - 만약 people[left] + people[right] <= limit 이면:
        - 두 명을 한 보트에 태울 수 있음 -> left 증가
    - 무거운 사람은 무조건 보트에 탐 -> right 감소, 보트 수(answer) 증가
4. 최종 보트 수 반환.

## 4. 깨달은 점 (Retrospective)

* 자료구조의 선택: deque는 양끝 요소 처리가 편하지만, 인덱스 접근만으로 충분한 경우 투 포인터가 메모리(O(1)) 측면에서 더 유리하다.
* 그리디의 증명: "가장 무거운 사람과 가장 가벼운 사람을 짝짓는 것이 항상 최선인가?"에 대해, 무거운 사람과 짝이 될 수 있는 가장 만만한 상대(가벼운 사람)를 먼저 소진시키는 것이 뒤에 남은 사람들의 선택지를 넓혀준다는 것을 이해함.
* 언어별 특성: Python의 간결함, Go의 명료한 메모리 제어, Java의 엄격한 타입과 성능 측정 도구(JMH)의 차이를 실전 코드로 비교해 봄.
