📝 TIL: 언어별 알고리즘 최적화 전략 (Target Number)1. 개요

* 문제: 프로그래머스 - 타겟 넘버
* 핵심: $2^N$개의 경우의 수를 탐색하는 BFS/DFS 알고리즘의 성능 비교 및 언어별 최적화 기법 습득.

------------------------------
2. Java: 자료구조와 참조 스왑✅ 핵심 최적화

* ArrayDeque vs LinkedList: 단순 큐/스택 목적이라면 캐시 효율이 좋은 ArrayDeque가 압도적임.
* 참조 스왑: 매번 요소를 꺼내 옮기지 않고, 새로운 리스트를 만든 후 참조 변수만 교체(=)하여 복사 비용 제거.

📊 Benchmark 결과

| 구현 방식 | 실행 속도 (Score) | GC 부하 (Time) | 비고 |
|---|---|---|---|
| Deque 기반 BFS | 27,918 us/op | 1,073 ms | 매번 poll/add 발생 |
| List 참조 스왑 BFS | 20,256 us/op | 913 ms | 약 27% 개선 |
| 재귀 기반 DFS | 2,614 us/op | 0 ms | 약 8배 개선, GC 발생 안 함 |

------------------------------
3. Go: 인터페이스와 슬라이스 최적화✅ 핵심 최적화

* Interface 오버헤드: interface와 struct를 사용한 추상화는 유연하지만, 알고리즘 풀이 시 힙 할당(Heap Alloc)을 유발하여 느려질 수 있음.
* Slice Pre-allocation: make([]int, 0, capacity)로 공간을 미리 확보하여 슬라이스 재할당(Resizing) 비용 제거.

📊 Benchmark 결과

| 구현 방식 | 실행 속도 (ns/op) | 메모리 (B/op) | 할당 횟수 (allocs/op) |
|---|---|---|---|
| Interface Queue BFS | 17,904,193 | 83,192,802 | 340 |
| Slice 최적화 BFS | 3,915,366 | 16,777,210 | 20 |

------------------------------
4. Python: 메모이제이션과 불변성✅ 핵심 최적화

* @lru_cache: 동일한 인자의 함수 호출 결과를 캐싱하여 중복 계산 제거.
* Hashable 인자: lru_cache는 내부적으로 딕셔너리를 사용하므로, 인자로 리스트(Mutable)가 아닌 튜플(Immutable)이나 숫자를 받아야 함.

📊 성능 인사이트

* 파이썬 재귀는 함수 호출 오버헤드가 크지만, lru_cache를 적용하면 중복 탐색을 막아 성능을 비약적으로 높일 수 있음.
* 실무에서는 메모리 누수를 방지하기 위해 maxsize 제한이나 외부 캐시(Redis)를 고려해야 함.

------------------------------
5. 최종 결론 및 교훈

    1. 재귀(DFS): 공간 복잡도가 $O(N)$으로 매우 효율적이나, 호출 깊이가 깊어지면 StackOverflow 위험이 있음. (Java/Go에서 뚜렷한 성능 우위)
    2. 반복문(BFS): 안전하지만 매 단계 상태를 저장할 메모리 할당 비용이 발생함. 이때 참조 스왑이나 공간 선할당으로 성능을 보완해야 함.
    3. 언어별 특성:
    * Java: GC 부하를 줄이는 것이 관건.
        * Go: 슬라이스를 직접 다루는 것이 가장 빠름.
        * Python: 내장 데코레이터와 라이브러리(itertools) 활용이 핵심.
