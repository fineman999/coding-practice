* 다리를 큐(Queue)로 형상화: 다리 길이를 큐의 크기로 잡고, 매초 pop과 push를 반복하며 시뮬레이션한다.
* 시간 계산 최적화: 모든 트럭이 빠져나갈 때까지 루프를 돌기보다, 마지막 트럭이 다리에 올라가는 순간 루프를 종료하고 + bridge_length를 더해 마무리하는 것이 효율적이다.
* 무게 총합 관리: sum() 함수를 매번 호출하면 $O(L)$이 추가되어 성능이 급격히 저하된다. totalWeight 변수를 따로 두어 관리하는 것이 핵심이다 ($O(1)$).

2. 🚀 언어별 최적화 포인트Go (Golang)

* Zero-allocation: make([]int, bridgeLength)로 고정 배열을 생성해 원형 큐(Circular Queue)처럼 인덱스(ptr)만 옮기며 재사용한다.
* Escape Analysis: 내부 함수(pop, push)가 외부 변수를 참조하면 힙(Heap)으로 탈출할 수 있다. 성능을 극한으로 뽑으려면 인라인(Inlined) 처리가 되도록 루프 내에 직접 로직을 작성한다.
* 결과: Apple M2 Pro 기준 약 6.3us/op, 0 allocs/op라는 압도적 성능 달성.

Java

* Deque vs Array: ArrayDeque<Integer>는 사용하기 편하지만 Integer 박싱(Boxing) 비용이 발생한다. JMH 벤치마크 점수를 높이려면 int[] 배열을 사용하는 것이 유리하다.
* JMH (Java Microbenchmark Harness): 자바의 성능을 측정할 때 System.currentTimeMillis() 대신 JMH를 사용하여 Warmup과 기댓값을 정확히 측정한다.
* ArrayDeque 활용: Deque를 써야 한다면 initialCapacity를 지정해 내부 배열 재할당을 방지한다.

Python

* deque 사용: 리스트의 pop(0)은 $O(N)$이므로, 반드시 collections.deque의 popleft()를 사용하여 $O(1)$로 처리한다.

3. 📊 성능 진단 도구 활용법

* Go: go test -bench . -benchmem으로 메모리 할당량 확인. -gcflags="-m"으로 컴파일러의 최적화 결정(Inlining, Escape) 확인.
* Java: JMH를 통해 ns/op 단위의 미세한 성능 차이 분석.
