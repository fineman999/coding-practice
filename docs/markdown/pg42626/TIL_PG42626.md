## 🐹 1. Go 언어 핵심 지식

### ① 컴파일 타임 인터페이스 검증 (Compile-time Verification)
```go
var _ QueueInterface[int] = (*PriorityQueue[int])(nil)
```
*   **`var _` (블랭크 식별자):** 메모리를 차지하지 않고 버려지는 변수 (컴파일러 잔소리 회피용)
*   **`QueueInterface[int]` (인터페이스 타입):** 검증하고자 하는 규격(자격증)
*   **`(*PriorityQueue[int])(nil)` (타입 변환):** 실제 객체를 생성하여 메모리를 낭비하지 않고, 빈 값(`nil`)에 구조체 껍데기(포인터 타입)만 씌워서 전달.
*   **목적:** 해당 구조체가 인터페이스의 모든 메서드를 구현했는지 **실행 전(컴파일 단계)에 100% 강제로 검사**하기 위함.

### ② 리시버(Receiver)의 선택: 포인터(`*`) vs 값
*   **포인터 리시버 `(pq *PriorityQueue)`:** 상태를 **수정**할 때, 구조가 무거울 때 사용.
*   **값 리시버 `(pq PriorityQueue)`:** 상태를 변경하지 않는 작고 가벼운 데이터(`Point`, `Time` 등)를 복사해서 넘길 때 사용.
*   🚨 **절대 원칙:** 한 구조체 안에서 두 리시버를 섞어 쓰지 마라! (인터페이스 호환성 문제 및 `Mutex` 등의 내부 상태 복사 버그 방지)

### ③ 메모리 할당의 주범: 박싱(Boxing)과 언박싱(Unboxing)
*   `container/heap`은 `any` 타입을 매개변수로 받음.
*   **박싱 (Boxing):** `int`(원시 타입) 👉 `any` 타입으로 들어갈 때 힙(Heap) 메모리에 상자를 새로 만듦 **(메모리 할당 발생 🚨)**
*   **언박싱 (Unboxing):** `any` 👉 `.(int)` 로 꺼낼 때는 기존 상자에서 값만 빼오므로 추가 할당 없음.
*   **제네릭과의 차이:** 제네릭은 런타임에 박싱을 하지 않고 컴파일 타임에 해당 타입 전용 코드를 생성하므로 성능이 압도적으로 좋음.

## 🐍 2. Python 핵심 지식

### ① `heapreplace`를 통한 극한의 최적화
```python
# ❌ 기존 방식: O(log N) 동작 2번 발생 (트리를 두 번 오르락내리락)
first = heapq.heappop(scoville)
heapq.heappush(scoville, mixed)

# ✅ 개선 방식: O(log N) 동작 1번 발생 (속도 10% 향상, 극강의 안정성)
first = heapq.heappop(scoville)
heapq.heapreplace(scoville, mixed) # 루트(가장 작은 값)를 빼면서 동시에 새 값을 꽂아 넣음!
```

## 📊 3. 성능 측정 (pytest-benchmark) 완벽 해독

### ① 스톱워치의 비밀: Rounds vs Iterations
*   **Iterations (연속 실행 횟수):** 함수가 너무 빨라서 측정이 안 될 때, 스톱워치를 한 번 켜고 끌 동안 **함수를 연속으로 묶어서 실행하는 횟수**.
*   **Rounds (반복 횟수):** 통계(평균, 표준편차)를 내기 위해 그 스톱워치 측정 세트를 **총 몇 번 반복했는지** 나타내는 횟수.

### ② 성능 통계 지표 분석
*   **StdDev (표준편차):** 평균(Mean)으로부터 각각의 기록이 평균적으로 얼마나 떨어져 있는가? **(작을수록 기복 없이 안정적임)**
*   **IQR (사분위 범위):** 운 좋게 빠른 상위 25%, 운 나쁘게 느린 하위 25%를 잘라내고 남은 **진짜배기 50% 기록들 간의 시간 격차**.
*   **Outliers (튀는 값):** `3;0` (왼쪽: 표준편차를 가볍게 벗어난 횟수 / 오른쪽: IQR 범위를 심각하게 뚫고 나간 비정상적인 일탈 횟수)
*   **OPS (초당 처리량):** `1초 ÷ 평균 실행 시간`. 초당 이 함수를 몇 번이나 실행할 수 있는지를 나타내는 체력 지표.
