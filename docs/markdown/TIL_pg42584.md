💡 1. 문제 핵심 요약

* 목표: 각 시점의 주식 가격이 담긴 배열을 보고, 가격이 떨어지지 않은 기간이 몇 초인지 구하기.
* 포인트: 가격이 떨어지더라도 떨어지는 그 순간까지의 1초는 기간에 포함됨.

------------------------------
🚀 2. 알고리즘 설계 (O(N^2) vs O(N))❌ 단순 이중 루프 ($O(N^2)$)

* 각 가격마다 뒤에 있는 모든 가격을 비교함.
* 데이터가 100,000개일 경우 최대 100억 번 연산이 발생하여 효율성 테스트 탈락 가능성 높음.

✅ 단조 스택(Monotonic Stack) 활용 ($O(N)$)

* 핵심 아이디어: "아직 가격이 떨어지지 않은 인덱스"를 스택에 보관함.
* 동작 원리:
1. 현재 가격이 스택 맨 위의 가격보다 낮다면(가격 하락), 스택에서 꺼내며 기간을 계산 (현재 시점 - 들어온 시점).
    2. 낮지 않다면 스택에 계속 쌓음.
    3. 루프 종료 후 스택에 남은 인덱스들은 끝까지 가격이 떨어지지 않은 것이므로 전체 시간 - 들어온 시점으로 계산.
* 복잡도: 모든 원소가 스택에 딱 한 번 push, 딱 한 번 pop 되므로 시간 복잡도는 $O(N)$.

------------------------------
🛠️ 3. 언어별 구현 및 최적화 포인트☕ Java

* 핵심 클래스: Stack 대신 ArrayDeque 사용 (동기화 오버헤드가 없어 성능 우위).
* 최적화: new ArrayDeque<>(n)와 같이 초기 용량을 지정하여 내부 배열 재할당(Resizing) 방지.
* 인터페이스: Deque<Integer> stack = new ArrayDeque<>();와 같이 인터페이스 타입을 사용하여 유연성 확보.

🐍 Python

* 파이토닉(Pythonic): answer를 리스트 컴프리헨션으로 미리 초기화([n - i - 1 for i in range(n)])하면 마지막 스택 처리 루프를 생략 가능.
* 슬라이싱: stack[-1]을 활용해 스택의 Top에 직관적으로 접근.

📊 4. 성능 측정 결과 (Benchmark)

* 환경: Apple M2 Pro
* 개선 효과:
* 이중 루프: 약 14초 ($14,034$ ms)
    * 스택($O(N)$): 약 0.35ms ($357$ μs)
    * 결과: 약 40,000배 이상의 성능 향상 달성.
    * 메모리: 초기 용량 설정 시 할당 횟수(allocs/op) 대폭 감소.

🧠 5. 오늘의 깨달음

1. 중첩 루프 내에서 불필요한 중복 탐색이 발생한다면 스택(Stack)이나 해시(Hash)를 먼저 떠올리자.
2. 자바에서 스택 구조가 필요할 땐 고민 없이 ArrayDeque를 선택하자.
3. 벤치마크 지표 중 ns/op는 시간, allocs/op는 메모리 효율을 나타내며, 미리 공간을 할당(Capacity)하는 것만으로도 성능이 크게 개선된다.
