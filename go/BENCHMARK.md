### 📊 Go (Golang) 벤치마크 결과표 분석 가이드 (`-benchmem`)

Go의 벤치마크는 파이썬처럼 복잡한 통계를 보여주지 않는 대신, 실무 서버 최적화의 핵심인 **"실행 속도"**와 **"메모리 할당량(GC 부하)"**을 아주 직관적으로 보여줍니다.

#### 1. 실행 환경 정보 (Environment)
```text
goos: darwin              # 운영체제 (macOS)
goarch: arm64             # CPU 아키텍처 (Apple Silicon)
pkg: .../pg86491          # 테스트가 실행된 패키지 경로
cpu: Apple M2 Pro         # 실제 코드가 돌아간 CPU 모델명
```

#### 2. 핵심 성능 지표 (Core Metrics)
가장 중요한 벤치마크 결과 줄에 대한 분석입니다.
> `BenchmarkSolution-10    170434    6359 ns/op    0 B/op    0 allocs/op`

| 항목 | 출력된 값 | 의미와 실무 해석 가이드 |
| :--- | :--- | :--- |
| **Name & Cores** | `BenchmarkSolution-10` | 벤치마크 함수 이름과 **사용된 CPU 코어 수(-10)**입니다. |
| **Iterations** | `170434` | 신뢰할 수 있는 평균을 구하기 위해, Go 프레임워크가 코드를 **17만 번 이상 반복 실행**했다는 뜻입니다. (파이썬의 Rounds와 동일) |
| **Speed (시간)** | `6359 ns/op` | ⭐��� 함수 1번 실행당 **평균 6,359 나노초(ns)**가 걸렸다는 뜻입니다.<br>*(참고: 6,359ns = 6.3us = 0.006ms. 파이썬(941us)보다 약 147배 빠른 압도적인 속도입니다!)* |
| **Memory (용량)** | `0 B/op` | ⭐️ 함수 1번 실행당 힙(Heap) 메모리를 **0 바이트** 사용했다는 뜻입니다. |
| **Memory (횟수)** | `0 allocs/op` | ⭐️ 함수 1번 실행당 메모리 할당 작업이 **0번** 일어났다는 뜻입니다. |

#### 3. 실무 판단 기준 (Good or Bad?)

Go 개발자들은 속도(`ns/op`)도 중요하게 생각하지만, 그 이상으로 **메모리 지표(`B/op`, `allocs/op`)에 엄청난 집착**을 보입니다.

* 🟢 **완벽한 상태 (Perfect): `0 allocs/op`**
    * 현재 작성하신 코드의 상태입니다!
    * 반복문 안에서 새로운 슬라이스(`[]int`)를 만들거나 문자열을 합치는(`+`) 등의 불필요한 메모리 할당이 전혀 없다는 뜻입니다.
    * **왜 좋은가?** 메모리 할당이 없으면, 찌꺼기 메모리를 청소하는 **가비지 컬렉터(GC)가 작동할 일이 없어져서 서버가 절대 멈칫거리지 않고 최상의 속도를 유지**합니다.

* 🔴 **개선이 필요한 상태 (Bad): 높은 `allocs/op`와 `B/op`**
    * 만약 결과가 `1200 B/op  10 allocs/op` 처럼 나온다면?
    * 함수가 한 번 실행될 때마다 메모리를 10번씩 새로 발급받고 있다는 뜻입니다. 초당 1만 번 호출되는 API 서버라면, 1초마다 메모리 할당이 10만 번 일어나서 서버가 금방 느려집니다.
    * **해결책:** 슬라이스 용량을 미리 지정(`make([]int, 0, capacity)`)하거나, 전역 객체를 재사용(`sync.Pool`)하여 메모리 할당(allocs)을 `0`에 가깝게 줄이는 리팩토링을 진행해야 합니다.
